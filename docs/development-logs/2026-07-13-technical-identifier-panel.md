# 开发日志:技术标识区域重排版(2026-07-13,批次 2/5)

## 本轮需求

用户反馈"对应业务内容"与"标识值"列在技术标识表格中发生重叠,阅读困难,并要求：确认技术标识仅管理员可见、默认收起;解决列宽不足和文字重叠;标识值支持正常换行、选择、复制,不逐字断行,不撑破页面,窄屏也正常。

## 问题原因(浏览器实测确认)

用真实管理员账号(密码由用户在对话中提供,登录后立即验证,未写入任何日志或代码)登录后打开付款详情页(CN"柴"、12 条明细、已撤销付款,`/admin/payments/47424003-deb7-40dd-93dc-548466d69ef3`),展开"查看技术标识"确认了两个真实问题:

1. **列宽不足导致溢出**:原表格用 `table-layout: fixed`,四列宽度里只给"标识类型"(120px)、"对应业务内容"(220px)、"操作"(90px)设置了显式宽度,唯独"对应业务内容"列本身没有 `overflow-wrap`/`word-break` 规则。当内容是形如 `IMP-63634a4de25b-233d38653b / 26感谢祭单领` 这种带长不可断行 token 的字符串时,在 220px 固定列宽下会横向溢出,visually 压到右边"标识值"列上,这正是用户描述的"重叠"。
2. **同一订单的技术标识被逐条重复展开**:这笔付款覆盖 12 条订单明细,但它们都属于同一个订单、同一个导入 Sheet。原实现按"每条明细展开一组标识"生成,导致"订单 ID""来源 Sheet"这类订单级别的字段被机械重复 12 次,总共渲染了 73 行,进一步加剧了拥挤和误读风险(用户在第 8 条里也明确提到了这一点)。

## 设计决策

没有采用"保留表格、只加宽某一列"的局部补丁,而是按用户在需求里给出的建议,把四列表格整体改成**两行卡片布局**:

- 第一行:标识类型(小徽章)+ 对应业务内容(允许换行,不再固定宽度)+ 复制按钮,一行内用 flex 布局,自动换行。
- 第二行:完整标识值,独占一行,等宽字体,`overflow-wrap: anywhere; word-break: normal`(只在必要处按 Unicode 断点换行,绝不逐字符断开),可选中、可复制。

同时新增 `mergeTechnicalIdentifiers()`:按"类型 + 值"去重,把指向同一个值的多个业务场景合并到一条记录里,把原来重复的"对应业务内容"合并成一个用"、"连接的列表(超过 3 项时显示"前 3 项 + 等 N 项",避免单行过长)。这样"减少无意义重复"的同时,"订单明细与商品之间的对应关系"依然可以在合并后的业务内容里看到完整列表,没有信息丢失。

这一改动直接复用了 5 个页面共用的 `TechnicalIdentifier` 数据结构和 `addTechnicalIdentifier` 生成逻辑,只改了渲染层和一个新增的合并函数,没有改动任何业务查询逻辑。

## 修改文件

- `frontend/src/App.vue`:
  - 新增 `mergeTechnicalIdentifiers()` 函数(紧邻 `addTechnicalIdentifier` 之后)。
  - 5 处技术标识渲染区块(付款详情、用户详情、订单详情、导入历史、导入详情)全部从 `<table>` 改为 `<div class="technical-list"><article class="technical-item">...` 卡片列表,并统一套上 `mergeTechnicalIdentifiers(...)`。
- `frontend/src/style.css`:删除旧的 `.technical-table`/`.technical-table table`/`.technical-table th/td:nth-child(...)`/`.technical-table code` 规则,新增 `.technical-list`、`.technical-item`、`.technical-item__head`、`.technical-item__type`、`.technical-item__context`、`.technical-item__value`。

## 意外发现与修复:文件被写入了一个空字节

编辑 `App.vue` 新增 `mergeTechnicalIdentifiers` 函数时,写入结果里 `row.type + ' ' + row.value` 这行的空格字符被写成了一个真实的 `\x00` 空字节(而不是空格),导致 `grep`/`vue-tsc` 一度把文件当成二进制文件。已经用脚本定位并替换回普通空格,替换后重新跑了完整构建确认恢复正常且没有引入其它变化。记录在此以防之后有人看到文件历史里这一处诡异改动感到困惑 —— 这是编辑过程中的工具层面异常,已现场修复,和业务逻辑无关。

## 权限影响

复核确认(未新增改动):

- 5 处技术标识区域全部位于 `admin-payment-detail`、`admin-user-detail`、`admin-order-detail`、`admin-import-history`、`admin-import-detail` 路由下,这些路由统一经过 `ensureAdmin()` 门禁,普通用户查询页 `/query` 的模板里完全没有技术标识相关标签。
- `<details>` 默认没有 `open` 属性,展开状态由用户点击驱动,刷新或首次进入时始终是收起状态。
