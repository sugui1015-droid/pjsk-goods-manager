## 阶段 2：迁移器行为离线分析

- 开始时间：2026-07-15 13:45（本机时间）

### 2.1 两个 `0005` 的准确执行顺序

`RunMigrations` 用 `fs.ReadDir` 读取全部 `*.sql`（排除目录），随后 `sort.Strings(names)` 按完整文件名做字节序排序。`0005_import_history.sql` 与 `0005_product_series.sql` 前 5 个字节相同（`0005_`），第 6 个字节 `i`(0x69) < `p`(0x70)，因此顺序恒定为：

1. `0005_import_history.sql`
2. `0005_product_series.sql`

两者均在 `0004_import_confirm.sql` 之后、`0007_import_revert.sql` 之前执行。该顺序由字节序决定，跨平台、跨文件系统一致（`embed.FS` 的 `ReadDir` 本身也按文件名排序，代码再显式排序一次，属双重保证）。

### 2.2 `schema_migrations.version` 记录的值

记录的是**完整文件名**（含 `.sql` 后缀），例如 `0005_import_history.sql`。既不是数字编号，也不是去后缀名。判断"已执行"用 `select exists(... where version = $1)` 精确匹配完整文件名；插入记录也用完整文件名。因此两个 `0005` 在 `schema_migrations` 里是两行互不相干的记录。

### 2.3 五种数据库状态下的行为

| 状态 | 行为 | 风险 |
| --- | --- | --- |
| 两个 `0005` 都未执行 | 依 2.1 顺序分别执行并各插入一行记录 | 无 |
| 只执行了其中一个 | 已执行者被精确文件名匹配跳过，另一个正常执行；不会误判 | 无 |
| 两个都已执行 | 两行记录均存在，两文件均跳过；不会重复执行 | 无 |
| 旧环境已执行到 `0018` | `0001`–`0018`（18 行）全部跳过，仅执行 `0019_admin_auth_audit_events.sql` | 无（`0019` 为纯新建表/索引，`if not exists` 且不触碰旧表） |
| 新空库 | 先建 `schema_migrations`，再按文件名顺序执行全部 19 个文件（`0001`…`0005×2`…`0007`…`0019`） | 无编号断层影响：缺 `0006` 只是文件名序列里没有这一项，迁移器不校验连续性 |

### 2.4 现有机制的风险盘点

- **排序风险：低。** 所有现有文件都是 4 位零填充前缀，字节序 = 数字序（至 `9999` 前恒成立）。风险仅在未来有人加入非 4 位前缀（如 `010_`、`0005a_`）或大小写不一致的文件名时出现。当前无自动防护（本阶段以离线测试补上）。
- **事务风险：低。** 每个迁移在独立事务中执行，SQL 与 `schema_migrations` 插入同事务提交，失败整体回滚——不存在"执行了但没记录"或"记录了但没执行"的中间态。PostgreSQL DDL 可回滚，现有 19 个文件均无不可入事务的语句（无 `create index concurrently` 等）。失败时记录文件名日志并中止启动，不会带病继续。
- **重复执行风险：仅当文件被重命名或 `schema_migrations` 行被手工删除时出现。** 两者均已被 HANDOVER §16 与本任务边界禁止。附带缓冲：两个 `0005` 的 SQL 本身是幂等写法（`add column if not exists` / `create index if not exists`），即使在极端误操作下重放也不会报错——但这是纵深防御，不是依赖项。
- **手工修改记录风险：** 向 `schema_migrations` 手工插入行会导致对应迁移被永久跳过（静默缺表/缺列），删除行会导致重放。禁止手工修改是硬边界。
- **附带发现（非编号问题）：** `backend/main.go` 用同一个 10 秒超时 `context` 覆盖"连接数据库 + 全部迁移"。对本地库无碍；若正式库是远程（如 Supabase），新空库首次启动要按序执行 19 个迁移、每个约 4–5 次网络往返，高延迟下可能触顶超时。中途超时是安全的（单迁移事务回滚，重启后从断点继续），但部署时应知晓"首启可能需要重启一次才能跑完全部迁移"。本阶段不改代码，登记为部署注意事项。

### 2.5 是否需要新的修复迁移

**不需要。** 理由：

1. 双 `0005` 造成的最终 schema 是正确的，两文件职责互不重叠（`import_batches` 加列建索引 vs `products` 加列），没有任何数据库状态需要修复。
2. 编号问题的本体是"文件名不满足规范"，而迁移文件无法修改历史文件名——任何新迁移都改变不了这一点。
3. 新增一个"空的占位 `0006`"或"完整性检查迁移"反而引入新风险：占位文件会让新旧库的 `schema_migrations` 行数出现无意义分叉；检查逻辑放进迁移意味着检查失败会阻断启动且难以回滚，与"迁移只做 schema 变更"的现有语义冲突。

若未来确有必要新增迁移（业务需要，而非编号修复），编号从 `0020` 顺延，永不复用已出现过的前缀，也不回填 `0006`。

### 2.6 不重命名、不删记录、不伪造记录的解决路径

采用"登记历史例外 + 离线测试锁定行为 + 部署前只读核对"三件套：

1. 在文档中把 `0005×2`、缺 `0006` 正式登记为**永久历史例外**（HANDOVER §14 已有说明，本阶段补充部署核对文档）。
2. 新增离线单元测试，锁定：按完整文件名排序、双 `0005` 分别加载、`0019` 可加载、嵌入 FS 与磁盘一致、除 `0005` 外前缀唯一且为 4 位数字、除 `0006` 外编号连续。未来任何人再引入重复前缀或坏文件名，`go test` 直接失败。
3. 正式部署前由人工执行只读 SQL 核对目标库 `schema_migrations` 实际状态（阶段 4 给出清单与分支决策），不猜测、不修改。

## 阶段 5：方案对比与结论

- 开始时间：2026-07-15 14:05（本机时间）
- 任务书列出的 5 个候选方案逐一评估：

| 方案 | 结论 | 理由 |
| --- | --- | --- |
| 1. 保持原样，只登记历史例外 | **采纳（作为基础）** | 唯一不产生任何运行时行为变化的方案；机制上双 `0005` 本就互不冲突。单独采用的缺点是"没有防再犯、没有部署核对"，故与 2 项补强合并。 |

| 3. 迁移器加重复前缀检测 + 白名单 | 否决（运行时形态），改为**测试时等价物** | 部署前夕改启动路径核心代码引入新风险：白名单配置错误会让现有正常库无法启动。改用离线单元测试对真实迁移目录做同样校验（除 `0005` 外前缀唯一、除 `0006` 外连续、命名规范），零运行时风险，`go test` 即拦截再犯。 |
